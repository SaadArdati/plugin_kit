Plugins are plain Dart classes, so most of what you do to test them is what you would do for any Dart class: construct, poke, assert. The parts that are specific to Plugin Kit are how to set up a registry, a bus, a context, or a whole runtime without spinning up your actual application.

Three common levels of test, from cheapest to most integrated.

This page is the verification pass for the guide path: after a feature is a
plugin, its behavior should be testable at the same level you expect it to
work.

## Stubbing the event bus

If your test target uses a real bus but you do not want to also test the bus, use a real `EventBus` as a stub. Nothing prevents you from emitting events "from outside" to drive whatever handler you are testing.

You rarely need to mock the bus itself. It is simple, fast, and well-exercised. Testing against a real `EventBus()` usually gives better signal than a mock would.

## Patterns that age well

- Prefer Level 2 (registry + bus + context) for most plugin tests. Fast, no runtime setup, and it catches real behavior.
- Use Level 3 for anything that involves settings reconciliation, enablement precedence, or multiple sessions.
- Keep a small `TrackingPlugin` helper in your test file for lifecycle assertions. You will reuse it.
- If a plugin depends on another plugin's service, register both in the test. Do not mock the service. The registry was designed for exactly this kind of swap.
