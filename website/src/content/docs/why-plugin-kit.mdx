---
title: Why Plugin Kit?
description: "When apps grow into platforms, the same plumbing keeps reappearing. Plugin Kit packages it once: real feature lifecycles, replaceable services, sealed sessions, typed events."
---

Plugin Kit reduces the manual lifecycle wiring and override branching that build up in apps that have grown into platforms: apps with real feature lifecycles, replaceable services, isolated sessions, and typed events between parts that have never been formally introduced.

When an app grows past a few features, the same patterns keep showing up:

- A feature wants its own lifecycle (set up, run, tear down) instead of leaning on `initState`.
- Two implementations want to compete for the same slot, and the loser should still be reachable as a fallback.
- A second document, tab, or chat needs its own state without leaking into the first.
- A settings dialog wants to enable, disable, or reprioritize features at runtime.
- Features need to coordinate without holding references to each other.
- You want to prototype a third-party service integration without disrupting the original third-party service.
- Your boss wants a new feature you hate, but you want to build it in a way that does not mess with the code you like.

Most teams build that runtime by accident. A DI container learns priorities. A feature flag service learns to disable subsystems. A controller learns request and response. After enough additions, a team has rebuilt a substantial slice of a plugin framework without noticing.

Plugin Kit packages all of that into one runtime with explicit rules. You write plugins; it does the choreography.

There are many ways to structure a Dart or Flutter app, and there is no single right answer. What matters is whether the structure helps you ship the kind of app you actually want to ship. If you are wiring features by hand and noticing the same scaffolding for the third time, Plugin Kit is built for that case.

The value shines when you want to develop features in complete isolation, systems that may collide with each other,
or features that can be turned on and off at runtime.

Worked example:

- A chat agent plugin ships a system prompt as an independent service inside the plugin.
- A teammate wants to iterate on the system prompt without touching the original. Instead of editing the existing service, the teammate writes a new plugin that claims the same registration slot at a higher priority. The teammate iterates in isolation; if the variant pays off, the plugin stays enabled.
- A non-coder on the team wants their own system-prompt experiment. They edit the JSON config through a settings page from `plugin_kit_dialog` and override the active system prompt at runtime.
- An end user disables the "experimental" system prompt because they prefer the original. The toggle takes effect live.
- The original system prompt is still registered the whole time, ready as a fallback whenever someone wants it back.

Plugin Kit is designed around three values:

- **Composable**: Features stack through priority, namespaces, and capability tags rather than through scattered `if`
statements. Override stops being a code smell, and "the current implementation of X" becomes something the runtime
answers instead of the host code guessing.

- **Isolated**: Sessions are sealed by default. Cross-scope routing is always explicit. Open ten documents, and they
do not leak into each other; close one and everything session-scoped cleans up on the way out.

- **Reconcilable**: Settings, priorities, and enablement converge at runtime without a restart. Hand the runtime a new
`RuntimeSettings` and it works out which plugins to detach, which to attach, and which to leave alone.

Overall, Plugin Kit makes feature lifecycles explicit, so you can compose, override, and reconfigure features without
writing teardown code by hand each time.

## Design decisions that follow from those values

**Domain-agnostic core.** The runtime knows nothing about Flutter, servers, editors, agents, or any specific UI framework. Client code extends `PluginContext`, `Plugin`, and `PluginService` to model domain concepts. The same runtime drops into a CLI, a server, or a Flutter app unchanged.

**Competing implementations are a first-class idea.** Most DI containers treat override as a special case. Plugin Kit treats it as the normal case. That is why priority is on the registration, not on the slot: the slot is just a name, and any plugin that registers there is a candidate to win.

**Metadata without instantiation.** Capabilities live on registration wrappers, so host code can discover what a service claims to support without paying construction cost. This is how dynamic settings UIs, extension manifests, and slot inspectors stay fast.

**Settings as reconciliation input, not instruction.** Handing the runtime a new `RuntimeSettings` is a statement of intent ("this is the world as it should be"), not a script. The runtime figures out what has to attach, detach, or reconfigure to converge on the new state. Plugin authors write lifecycle hooks; they do not write "on settings change" handlers for every configurable knob.

**Errors surface.** The bus does not swallow handler exceptions. The runtime's lifecycle phases aggregate failures and throw `PluginLifecycleException`, naming the phase that failed. Silent failure in a plugin system is a debugging nightmare a year later.
