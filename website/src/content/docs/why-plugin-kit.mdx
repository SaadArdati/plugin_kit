---
title: Why Plugin Kit?
description: A short take on what Plugin Kit is for, the symptoms that motivate it, and the design values behind it.
---

Plugin Kit makes it easier to ship apps that have grown into platforms: apps with real feature lifecycles,
replaceable services, isolated sessions, and typed events between parts that have never been formally introduced.

When an app grows past a few features, the same primitives keep showing up:

- A feature wants its own lifecycle (set up, run, tear down) instead of leaning on `initState`.
- Two implementations want to compete for the same slot, and the loser should still be reachable as a fallback.
- A second document, tab, or chat needs its own state without leaking into the first.
- A settings dialog wants to enable, disable, or reprioritize features at runtime.
- Features need to coordinate without holding references to each other.
- You want to prototype a third-party service integration without disrupting the original third-party service.
- Your boss wants a new feature you hate, but you want to build it in a way that does not mess with the code you like.

Most teams build that runtime by accident. A DI container learns priorities. A feature flag service learns to
disable subsystems. A controller learns request and response. Somewhere around the fourth piece, a team has shipped
roughly forty percent of a plugin framework without noticing.

Plugin Kit packages all of that into one runtime with explicit rules. You write plugins; it does the choreography.

There are many ways to structure a Dart or Flutter app, and there is no single right answer. What matters is whether
the structure helps you ship the kind of app you actually want to ship. If you are wiring features by hand and noticing
the same scaffolding for the third time, this library is probably worth a look.

The value shines when you want to develop features in complete isolation, systems that may collide with each other,
or features that can be turned on and off at runtime.

Example:
- I built the chat agent with the system prompt as an independent service piece of the chat agent plugin.
- My colleague said he had an idea to improve the system prompt. Instead of modifying my system prompt destructively,
he builds a new plugin that claims the same registration slot with a higher priority. He can test and iterate on his
version without touching mine. If we end up enjoying it, we keep the plugin enabled!
- My Boss then says he has a fun system prompt idea he wants to experiment with, but doesn't write code and needs a UI
to modify it. He creates the json config via a nice settings page from `plugin_kit_dialog` and applies it to
override the default system prompt dynamically.
- One of our app users comes along and disables the "experimental" system prompt toggle because they prefer the original.
- Everyone is happy, and the original system prompt is still there as a fallback if we want to go back to it.

Plugin Kit is designed around three values:

- **Composable**: Features stack through priority, namespaces, and capability tags rather than through scattered `if`
statements. Override stops being a code smell, and "the current implementation of X" becomes something the runtime
answers instead of the host code guessing.

- **Isolated**: Sessions are sealed by default. Cross-scope routing is always explicit. Open ten documents, and they
do not leak into each other; close one and everything session-scoped cleans up on the way out.

- **Reconcilable**: Settings, priorities, and enablement converge at runtime without a restart. Hand the runtime a new
`RuntimeSettings` and it works out which plugins to detach, which to attach, and which to leave alone.

Overall, Plugin Kit makes feature lifecycles explicit, so you can compose, override, and reconfigure features without
writing teardown code by hand each time.
