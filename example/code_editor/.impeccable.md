# Design Context — code_editor example

## Users

Flutter developers evaluating `plugin_kit` and `plugin_kit_dialog`. They open this example to see what a non-trivial plugin host looks like in practice, and to judge whether the abstractions feel right. Their job: read the shell once, understand the contribution model, and decide whether to build on it.

## Brand Personality

Three words: precise, tool-like, calm. The example should feel like a real IDE someone could imagine using, not a styled demo. Voice is engineering-credible. The interface earns trust by looking like the tools the audience already lives in (JetBrains IDEs).

## Aesthetic Direction

Reference: **JetBrains "Islands Dark"** (New UI). Tool windows ("islands") sit on a near-black canvas, separated by thin canvas-colored gutters that produce a floating effect — no shadows, no glow. Each island has a slightly lighter fill (~#2B2D30) over the canvas (~#1E1F22), small rounded corners (~8px), and a hair-thin border that almost disappears.

Selection is a filled rounded rectangle in JetBrains blue (~#3574F0). Editor tabs use a thin top stripe in accent; everything else is a fill. Iconography is muted gray, accent blue when active. Typography is dense: 12–13px for chrome (Inter / system), 13px monospace in the code area, 28–34px row heights.

Anti-references:
- The current `code_editor` shell (bright blue status bar, inconsistent panel chrome, oversized chips).
- Generic Material 3 "productivity app" defaults (too roomy, too rounded, not IDE-flavored).
- Any visual that looks bespoke or hand-rolled.

The **AI Assist panel** is the only intentional exception: its island chrome matches the rest of the app, but the panel *interior* expresses a third-party AI provider's brand identity (e.g., Gemini gradients, accent shimmer). This is a deliberate cue that AI surfaces are vendor surfaces.

The `plugin_kit_dialog` opened from this app is a separate package with its own theme and stays visually distinct on purpose. The host theme does not import or extend the dialog's theme.

## Design Principles

1. **Stock Material only.** No bespoke widgets, no `EditorColors` / `EditorTextStyles` / `EditorDecorations` static classes, no extension methods, no theme getters. Every visual decision lives in a single `ThemeData` and propagates via `Theme.of(context)`. If a stock Material widget cannot produce the look, accept the closest stock approximation.
2. **One ThemeData, no parallel palettes.** Colors come from `colorScheme`. Typography comes from `textTheme`. Component shapes come from component themes (`cardTheme`, `tabBarTheme`, `chipTheme`, etc.). A widget that needs a color or font reaches into `Theme.of(context)` — never into a static class.
3. **Islands separated by gutters, not borders.** Visual hierarchy comes from the canvas-colored gap between panels and the lighter surface fill inside them. Borders are vestigial.
4. **Density matches an IDE, not a phone app.** Chrome rows 28–34px, panel headers small, padding minimal. Built from stock widgets sized down via theme defaults — not custom widgets.
5. **AI Assist is the branded exception.** Treat its panel interior as a "vendor surface" — Gemini-style gradients, brand accent — while keeping its island wrapper identical to every other panel.

## Tactical Notes

- Status bar: canvas-colored strip with muted small text. Not the accent color.
- Tabs: use stock `TabBar` with `indicatorWeight` 2 and `indicator` color = `colorScheme.primary` (top stripe via `TabBarTheme` if needed).
- Panel chrome: stock `Card` with `clipBehavior: antiAlias`, theme-defined radius, internal `ListTile` / `Row` for the header.
- Chips: stock `FilterChip` with theme defaults.
- Icons: stock `IconButton` (or `IconButton.outlined` for emphasis).
- Close buttons on panels: stock `IconButton` with `Icons.close` at 16px; rely on the theme's `iconButtonTheme` for sizing.
